## はじめに

```python
# Python: OK
fruits = [
    "apple",
    "banana",
    "cherry",  # ← この末尾カンマ、付けていい
]
```

```json
// JSON: NG
{
  "apple": 1,
  "banana": 2,
  "cherry": 3,  // ← これがあるとパースエラー
}
```

同じように見える書き方なのに、片方は許され、片方は許されない。本記事では **末尾カンマ (trailing comma)** がなぜ言語ごとに違う扱いを受けるのかを、設計思想に踏み込んで整理します。

## 言語別の対応一覧

| JSON | ❌ 不許可 | RFC 8259 で明示的に禁止 |
| JSONC | ✅ 許可 | VS Code 設定ファイル等で使用 |

| JavaScript (配列) | ✅ 許可 | 古来から |

| JavaScript (関数引数) | ✅ 許可 | ES2017 から |

## なぜこんな違いが生まれたのか

<!-- 採否判断: ここが本記事の核。
     設計思想の違いを Crockford 自身の言葉で示す。 -->

末尾カンマが許可されるかどうかは、**「人間の編集しやすさ」と「パーサのシンプルさ」のどちらを優先するか** という設計判断に帰着します。

### JSON の場合 — 最小主義の徹底

JSON は Douglas Crockford が 2001年に提案した、JavaScript のオブジェクト記法をベースにしたデータ交換フォーマットです [^json]。彼は仕様の **最小化** を徹底しました。

[^json]: RFC 8259 — https://datatracker.ietf.org/doc/html/rfc8259

JSON にコメント機能が無い理由について、Crockford は 2012年に有名な発言を残しています：

> I removed comments from JSON because I saw people were using them to hold parsing directives, a practice which would have destroyed interoperability.

（人々がコメントをパース指示を入れる用途に使い始めたのを見て、コメントを削除した。あれを残せば相互運用性が破壊されていた）[^crockford]

[^crockford]: Wikipedia "JSON" の記事内に記録されている、Crockford の Google+ 投稿 (2012) からの引用。

末尾カンマも同じ思想で扱われています。**書き手の便宜 < パーサ実装の単純さ・相互運用性**。あらゆるプログラミング言語で同じ JSON 文字列が同じデータに復元されるためには、文法は最も厳しく、曖昧さの余地を残さない方が良い、という哲学です。

### Python の場合 — 人間の編集を最適化

対照的に Python の公式 FAQ [^pyfaq] には、末尾カンマを許可する理由として明確に書かれています：

[^pyfaq]: Python 3 Design and History FAQ — https://docs.python.org/3/faq/design.html

> When you have a literal value for a list, tuple, or dictionary spread across multiple lines, it's easier to add more elements because you don't have to remember to add a comma to the previous line. The lines can also be reordered without creating a syntax error.

（複数行のリテラルでは、要素を追加するときに前の行にカンマを足す必要が無く、行の並び替えも文法エラーを起こさない）

つまり Python は **書き手の編集効率** を優先しています。プログラムは何度も編集されるものであり、末尾カンマを許せば編集ミスが減る、という判断です。

### JavaScript の場合 — 後付けの修正

JavaScript は当初、配列リテラルでは末尾カンマを許していましたが、**オブジェクトリテラル** では許していませんでした。これが ES5 (2009) で改められ、関数引数についても ES2017 で末尾カンマが許可されました [^mdn]。

[^mdn]: MDN: Trailing commas — https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Trailing_commas

歴史的には Internet Explorer の JScript が `[1, 2, 3,]` を `[1, 2, 3, undefined]` と解釈する独自バグを抱えていた時代があり、ブラウザ間の挙動差を生んだ汚点とも言えます。現代の JavaScript では完全に標準化されています。

## Git diff で見ると差が劇的

<!-- 採否判断: ここが実用的価値の山場。
     この実例は既存記事でも書かれているが、diff の出力を実際に見せる記事は少ない。
     具体性で差別化。 -->

末尾カンマの実用的なメリットは、**Git diff のノイズが減る** ことです。実例を見るのが早いです。

### 末尾カンマが無い場合（JSON 風）

```diff
 {
   "apple": 1,
   "banana": 2,
-  "cherry": 3
+  "cherry": 3,
+  "date": 4
 }
```

`cherry` の行に **カンマを足す変更** と、`date` の行を **追加する変更** の2行 diff です。`cherry` 自体は値も意味も変わっていないのに、変更履歴に出てきてしまう。

### 末尾カンマがある場合（Python 風）

```diff
 fruits = {
     "apple": 1,
     "banana": 2,
     "cherry": 3,
+    "date": 4,
 }
```

`date` の追加だけが diff に現れます。**意図した変更だけが履歴に残る**。

これが特にレビューで効いてきます。コードレビュー時に「この cherry の変更って何のため？」と一瞬考える時間が、チーム規模で積もると無視できなくなります。

## Go と Rust は積極派

Go は更に進んで、**複数行リテラルでは末尾カンマを必須** にしています。

```go
fruits := []string{
    "apple",
    "banana",
    "cherry", // ← 無いとコンパイルエラー
}
```

これは「文法の曖昧さを排除しつつ、編集効率を強制する」という Go らしい設計判断です。`gofmt` が常にこの形に整形するので、流派の議論自体が起きません。

Rust も `rustfmt` のデフォルトは末尾カンマを付ける派です。チーム内のスタイルが自動的に揃うため、これも議論の必要が無くなる仕組みです。

## C 言語の事情

<!-- 採否判断: 組み込み読者向けに C の話を入れるか。
     [選択肢A] 入れる（晴貴さん向き）
     [選択肢B] 削除

C 言語でも末尾カンマは許可されており、配列初期化と enum で使えます。

```c
int fruits[] = {
    1,
    2,
    3,  // OK
};

enum color {
    RED,
    GREEN,
    BLUE,  // C89 から OK
};
```

実は C89 では enum の末尾カンマは厳密には禁止されていましたが、C99 で正式に許可されました [^c99]。多くのコンパイラは C89 モードでも extension として受け入れていたので、実質的には C89 時代から書かれてきました。

[^c99]: ISO/IEC 9899:1999 (C99) Section 6.7.2.2 で enum の末尾カンマが許可された。

組み込みでよく使われる MISRA C のような厳格なコーディング規約では、末尾カンマの使用について規約ごとに方針が異なります。MISRA-C:2012 では明示的な禁止ルールはありませんが、レビュー時の慣習として削除を求めるチームもあります。

## JSONC, JSON5 — 妥協点を求めた派生規格

<!-- 採否判断: JSONC / JSON5 の話を入れるか。
     [選択肢A] 入れる（VS Code を使う読者には実用的）
     [選択肢B] 削除
     現案: A -->

JSON の最小主義は素晴らしい設計ですが、**設定ファイル** として使うときは不便です。コメントが書けず、末尾カンマも許されないため、編集ミスでパースが壊れやすい。

そこで派生規格が生まれました：

- **JSONC** (JSON with Comments): VS Code の `settings.json` で採用。`//` と `/* */` のコメントと末尾カンマを許可
- **JSON5**: より緩く、シングルクオート、末尾カンマ、コメント、識別子のクオート省略などを許可
- **HJSON**: 人間が書きやすいことを目指した、もっと緩いフォーマット

これらはどれも「設定ファイル用には JSON は厳しすぎる」という現場の不満から生まれました。**API のレスポンスには JSON、ローカル設定ファイルには JSONC**、という使い分けが現実的です。

## まとめ — 設計思想が生んだ小さな分水嶺

末尾カンマの許否は、わずか1文字の違いですが、その背後には **「誰のための文法か」** という設計思想の違いがあります。

- **データ交換フォーマット** は、機械が確実に同じ意味で読めることを優先する → 末尾カンマ禁止
- **プログラミング言語** は、人間が編集しやすいことを優先する → 末尾カンマ許可
- **設定ファイル用の中間フォーマット** (JSONC, JSON5, TOML) は人間寄りに振る → 許可

迷ったら以下の指針で考えると整理しやすいです：

- **書いているのが API のレスポンスや永続化フォーマット**: 厳格に、末尾カンマなし
- **書いているのが人間が編集するコードや設定**: 末尾カンマあり
- **チームで揃えたいなら**: フォーマッタ (Prettier, Black, gofmt, rustfmt) に任せる

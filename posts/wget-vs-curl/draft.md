最近、Dockerfile を生成 AI に書いてもらうことが増えました。便利なのですが、出力された `RUN` 行を眺めていると、`wget` と `curl` がほぼ毎回顔を出すのに気づきます。

```dockerfile

    curl \
    wget \
    ca-certificates \
 && rm -rf /var/lib/apt/lists/*
```

「両方とも入れておけばだいたい何とかなる」のは知っているのですが、改めて聞かれると **何が違うのか答えられない**。AI が書いてくれたものをそのままコピペしてしまっているだけで、自分で選んでいない。これは良くない、と思って調べたのが本記事の出発点です。

調べてみると、両者は単に「似たツール」ではなく、設計思想・歴史・実装範囲のすべてで明確に違っていました。本記事ではその違いを整理します。

## 設計思想の違い

<!-- 採否判断: ここがこの記事の核。これを書かないと「wget vs curl」記事にならない。
     必須セクション。 -->

### wget — Unix の `cp` のように

wget は **ファイルを取りに行く専用ツール** です。コマンド名の "wget" は "web get" の略で、そのまま「ウェブからファイルを取ってくる」用途に特化しています。設計思想を Daniel Stenberg は「Unix の `cp` コマンドに近い」と表現しています。`cp` がファイルをコピーする道具であるように、wget は URL からローカルへのコピーを担う道具です。

最大の特徴は **再帰的ダウンロード**。`-r` オプションで指定 URL からリンクを辿り、サイト全体をミラーリングできます。

```bash
# 例: サイト全体をオフライン保存
wget --mirror --convert-links --adjust-extension --page-requisites \
     --no-parent https://example.com/
```

### curl — Unix の `cat` のように

curl は **データ転送のための汎用ツール** です。"curl" は "Client URL" の略。Stenberg はこちらを「Unix の `cat` に近い」と説明しています。標準入出力を介してデータを流すパイプ指向の道具です。

```bash
# 例: API を叩いてレスポンスを jq に流す
curl -s https://api.example.com/users | jq '.[].name'
```

curl のもう一つの大きな特徴は、コマンドラインツールとは別に **`libcurl`** というライブラリを提供している点です。これが後述する組み込み機器への普及を可能にしました。

## libcurl はあなたの身の回りで動いている

curl の作者 Daniel Stenberg 自身の見積もりによれば、libcurl は世界に **数百億台** の規模でインストールされています [^1]。これは大げさな表現ではなく、以下のような場所で実際に動いています。

- **OS**: Windows 10/11 と macOS にプリインストール、iOS と Android、Chrome OS の標準コンポーネント
- **ゲーム機**: PlayStation、Xbox、Nintendo Switch
- **スマートカー**: 主要メーカーの車載インフォテインメントシステム（マップ更新、リモート診断などに使用）
- **アプリケーション**: YouTube、Skype、Instagram、Spotify、Netflix がそれぞれ独自に curl をバンドル
- **宇宙**: NASA の火星探査機にも搭載（Stenberg は冗談混じりに「curl は2つの惑星で動いている」と語っている）

[^1]: https://daniel.haxx.se/ — Stenberg 自身が "tens of billions of devices" と公表。

## ライセンスの違いが生んだ採用範囲

wget は GPL v3、curl は MIT です。GPL v3 はソースコード開示義務があるため、クローズドソースの製品にバンドルすることが難しく、組み込み機器の世界では curl の方が圧倒的に採用されやすい。これも libcurl がここまで普及した一因です。

## Wget の意外な特徴

<!-- 採否判断: 「curl の方がすごい」一辺倒だと記事が偏る。
     wget の良さもフェアに書く。 -->

ここまで curl 寄りの話が続きましたが、wget にも明確な強みがあります。

- **デフォルトが親切**: `wget URL` だけでファイルがローカルに保存される（curl は `-O` が必要）
- **不安定な回線での再開機能**: `-c` オプションで中断したダウンロードを再開できる。これは curl にも `-C -` があるが、wget の方が再開挙動が堅牢と評価されている
- **再帰ダウンロードは唯一無二**: curl にはこの機能がない
- **左手だけでタイプできる**: QWERTY キーボードで w-g-e-t は全て左手の範囲（地味に便利）[^2]

[^2]: 公式ページ https://daniel.haxx.se/docs/curl-vs-wget.html にも書かれている、Stenberg公認のジョーク的事実。

サイト丸ごとアーカイブしたい、不安定な接続で巨大ファイルを取りたい、というケースでは wget の方が向いています。

## PowerShell の `curl` は curl ではない

<!-- 採否判断: この罠の解説は他記事でも触れられているが、
     「なぜ Microsoft はこの設計にしたのか」「Windows 10 以降どう変わったか」
     まで深掘りしている記事は少ない。差別化ポイント。 -->

Windows ユーザーが踏みがちな罠です。PowerShell で `curl` と打つと、`Invoke-WebRequest` というコマンドレットが起動します。これは curl とは全く別物で、引数の体系も出力形式も違います。

```powershell
# これは curl ではなく Invoke-WebRequest が動く
curl https://example.com
```

理由は PowerShell の **エイリアス機能**。Unix 系コマンドからの移行を楽にするため、`curl` は `Invoke-WebRequest` のエイリアスとして登録されていました。

ただし Windows 10 (1803, 2018年4月) 以降、本物の `curl.exe` が `C:\Windows\System32\curl.exe` に同梱されています。PowerShell でも本物の curl を使いたければ、`curl.exe` と拡張子付きで呼ぶか、エイリアスを削除します。

```powershell
# 本物を呼ぶ
curl.exe https://example.com

# エイリアスを削除（恒久的にしたいなら $PROFILE に書く）
Remove-Item Alias:curl
```

PowerShell 6（Core、2018年）以降ではこのエイリアスは削除されており、`curl` はデフォルトで本物の curl.exe を呼びます。Windows PowerShell 5.1（OS同梱版）だけが今も古い挙動を引きずっています。

## 最近の動向 — wcurl と Wget2

<!-- 採否判断: ほとんどの既存日本語記事が触れていない最新動向。
     差別化のために入れたい。
     wcurl は 2024年に登場し、curl 8.14.0 (2025年5月) でバンドル化された。
     Wget2 は 2018年初版リリースだが日本語記事ではマイナー。 -->

### wcurl — curl プロジェクトが作った「wget 風 curl」

curl は便利な反面、`-o` や `-O` を付けないとファイルとして保存されないなど、初心者には不親切な面があります。これを解決するため curl プロジェクト自身が **wcurl** という薄いラッパーを 2024 年にリリースしました [^3]。`wcurl URL` だけでファイル保存ができます。curl 8.14.0 以降は curl のリリース tarball にバンドルされています。

[^3]: https://curl.se/wcurl

### Wget2 — wget の次世代版

GNU プロジェクトでは **Wget2** が wget の後継として開発されています [^4]。HTTP/2 対応、並列ダウンロード、libwget としてのライブラリ提供など、現代的な機能を備えています。ディストリビューションによっては既にパッケージ化されているので、`apt install wget2` などで試せます。

[^4]: https://gitlab.com/gnuwget/wget2

## まとめ

<!-- 採否判断: まとめは1段落でも箇条書きでもよい。
     [選択肢A] 用途別に「こういうときはこっち」型
     [選択肢B] 思想を再掲する型
     現案: A（読者にとって実用的） -->

雑にまとめると以下のような使い分けになります。

- **API を叩く / 単発のリクエストを試す**: curl
- **サイト全体をオフライン保存**: wget
- **不安定な回線で巨大ファイルをダウンロード**: wget
- **スクリプトに組み込む / ライブラリとして使う**: curl (libcurl)
- **Windows でどっちかしか入っていない**: 多くの場合 curl（プリインストール）

両者は競合というより補完関係にあります。Stenberg 自身、curl 公式ページに「`curl` という名前を使った wget の互換クローンが BusyBox にある」と書いていますが、これは皮肉ではなく事実の記述です。組み込み Linux の世界では BusyBox の wget で十分なことが多く、libcurl をフルで載せるのは贅沢な選択になります。
